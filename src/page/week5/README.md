# Week 5: **React & Next.js 신기능 – 동시성, Hooks 심화, Next 13 특징**

**학습 개요:** 이번 주에는 **React 18+와 Next.js 13+의 최신 생태계**를 학습합니다. 먼저 React 18에서 도입된 **Concurrency (동시성) 기능**을 이해하세요. **`useTransition`** 훅과 **Concurrent Rendering** 덕분에 React가 렌더링을 중간에 끊고 다른 업데이트를 끼워 넣을 수 있게 되었는데[react.dev](https://react.dev/reference/react/useTransition#:~:text=useTransition%20,blocking%20transitions), 이러한 작동 방식이 UI 응답성을 개선함을 배웁니다. 또한 React 18의 **자동 배치(Automatic Batching)**, **`useDeferredValue`** 등도 함께 보세요. React 19 버전(프리뷰 단계)에서는 **`use`**, **`useOptimistic`** 등의 신규 훅이 논의되고 있으니, React 공식 RFC나 블로그를 통해 **서버 컴포넌트**와 **`use`** 훅이 어떤 역할인지 알아둡니다. Next.js 13에서는 **App Router**가 도입되어 폴더 기반의 서버컴포넌트 라우팅, **Layout** 및 **Loading** 구성, **평행 라우팅(Parallel Routes)**과 **경로 인터셉트(Intercepting Routes)** 같은 혁신적인 라우팅 전략이 추가되었습니다[youtube.com](https://www.youtube.com/watch?v=RG85NeRQZCc#:~:text=Official%20Update%3A%20https%3A%2F%2Fnextjs.org%2Fblog%2Fnext,com%2Fshadcn%2Fstatus)[nextjs.org](https://nextjs.org/docs/app/api-reference/file-conventions/intercepting-routes#:~:text=Intercepting%20routes%20allows%20you%20to,routing%20paradigm%20can%20be). 이러한 기능들이 **모달 라우팅**이나 **멀티 패널 UI**를 가능케 한다는 것을 Next 공식 블로그를 통해 학습하세요. 또한 **Next.js의 데이터 패칭 전략**들을 정리합니다 – getServerSideProps(SSR), getStaticProps(ISR 포함), 그리고 Next 13의 **`fetch`** 사용 (**`fetch`**가 자동 캐시/재검증되는 메커니즘) 등을 비교합니다. 이어서 **클라이언트 상태 패칭**을 위한 **Axios vs React Query vs SWR** 접근을 이해합니다. 예컨대, Axios는 단순 fetch이고, React Query는 **캐싱/리패칭 제어를 포함한 도구**, SWR은 **Stale-While-Revalidate 패턴**으로 백그라운드 데이터 갱신을 해주는 도구입니다[dev.to](https://dev.to/leticiabytes/swr-vs-react-query-5el0#:~:text=SWR%20is%20a%20smaller%20library,that%20provides%20many%20more%20features). 각 접근의 장단점을 간략히 정리해보세요. 마지막으로 Next.js에서 자주 고려되는 **SEO 설정 (Head 관리)**, **이미지 최적화** (next/image), **폰트 최적화** (next/font or **`<link rel="preload">`**) 그리고 **GDPR 대응 글로벌 설정**(13장에 등장) 등을 개략적으로 훑어봅니다. 이번 주는 개념이 많으므로, Next.js 공식 문서의 **업그레이드 가이드**나 **블로그 글**을 따라 실습 프로젝트를 12 -> 13 버전으로 마이그레이션해보는 것도 도움이 됩니다.

- **React 18 동시성 (Concurrent Mode):** React 18에서는 **Concurrent Rendering**이 기본 도입되어, **`startTransition`** API로 낮은 우선순위 업데이트를 배경 처리할 수 있습니다. **`useTransition()`** 훅을 써보면서, 상태 업데이트를 transition으로 래핑하면 React가 그 업데이트를 병렬 처리하여 UI 응답성을 향상시키는 것을 실험해보세요[react.dev](https://react.dev/reference/react/useTransition#:~:text=useTransition%20,blocking%20transitions). 예를 들어 대량 리스트 필터링 시, 입력값 업데이트는 즉각 처리하되 리스트 필터링은 **`startTransition`**으로 감싸 배경에서 처리하게 할 수 있습니다. 또 **자동 배치**로 인해 setTimeout 등 이벤트 핸들러 밖에서도 여러 setState가 하나로 묶여 렌더링이 한 번만 일어나는 동작을 확인합니다. React 18의 새로운 훅 **`useId`** (동시성 환경에서도 고유 ID 보장)도 살펴보고, SSR hydration 변화를 이해합니다. 이 모든 기능은 React 공식 블로그 *“Introducing 18”* 글에 잘 정리되어 있습니다.
- **React 19 및 실험적 훅들:** React 19 버전(만약 2025년 말에 출시되었을 경우)에는 몇 가지 실험적 훅이 소개되었습니다. **`useOptimistic`** 훅은 Optimistic UI 업데이트를 간소화하고, **`useFormStatus`** 훅은 **`<form>`**의 서브밋 상태 등을 읽게 해줍니다. 가장 큰 변화는 **React Server Components (RSC)**의 실용화입니다. Next.js 13 App Router를 통해 이미 사용 중인 RSC 개념 – **서버에서 React 컴포넌트 렌더링하여 HTML과 데이터 일체를 보내는 방식** – 을 다시 한번 복습하세요. 그리고 React 19에서 새롭게 논의되는 **`use` 훅**은 서버 컴포넌트에서 Promise를 처리하기 위한 문법으로, **`await`**을 JSX에서 사용할 수 있게 하는 아이디어입니다. 공식 RFC나 Andrew Clark 등의 논의를 읽고, 서버 측 데이터를 한꺼번에 fetching 하면서도 컴포넌트 단위로 분리하는 패턴을 이해합니다. (이 부분은 다소 난이도가 있으므로 개념만 잡아두세요.)
- **Next.js 13 App Router와 라우팅 혁신:** Next.js 13의 가장 큰 변화는 기존 Pages Router에서 **App Router**로 전환된 것입니다. 폴더 구조 기반으로 **`layout.js`**, **`page.js`**, **`loading.js`**, **`error.js`** 등을 구성하여 계층적 UI를 선언하게 됩니다. 또한 **Parallel Routes**와 **Intercepting Routes**로 불리는 기능을 통해 한 페이지에서 다중 하위 라우트를 병렬로 렌더링하거나, 특정 링크를 클릭했을 때 **현재 페이지 레이아웃을 유지한 채** 새로운 페이지를 모달 등으로 띄우는 게 가능합니다[nextjs.org](https://nextjs.org/docs/app/api-reference/file-conventions/intercepting-routes#:~:text=Intercepting%20routes%20allows%20you%20to,routing%20paradigm%20can%20be)[youtube.com](https://www.youtube.com/watch?v=RG85NeRQZCc#:~:text=Official%20Update%3A%20https%3A%2F%2Fnextjs.org%2Fblog%2Fnext,com%2Fshadcn%2Fstatus). 공식 문서에 따르면 *“Intercepting routes allows loading a route from another part of your app within the current layout”*라고 설명되어 있습니다[nextjs.org](https://nextjs.org/docs/app/api-reference/file-conventions/intercepting-routes#:~:text=Intercepting%20routes%20allows%20you%20to,routing%20paradigm%20can%20be). 예를 들어 **`/photos`** 리스트 페이지에서 사진을 클릭하면 **`/photos/[id]`** 페이지를 전체 내비게이션 없이 **모달로 현재 페이지 위에 표시**할 수 있습니다. Next.js 13.3 블로그에 있는 예제를 한번 따라해보세요. 이런 병렬/인터셉트 라우팅은 **사용자 경험 향상**에 큰 도움이 됩니다 (모달 띄울 때 배경 리스트 유지 등). 또한 Next 13에서는 **Route Groups** (괄호로 묶은 디렉토리로 URL 경로에 영향을 주지 않는 레이아웃 그룹) 등의 개념도 있으니, 라우팅 챕터를 꼼꼼히 읽어봅니다.
- **Next.js의 데이터 패칭 전략 정리:** Next.js는 다양한 데이터 패칭 방법을 제공합니다. **SSR (getServerSideProps)**는 매 요청마다 서버에서 렌더링하고, **정적 생성(SSG)**은 빌드 시 한번 페이지를 생성합니다. **ISR (Incremental Static Regeneration)**은 정적 페이지를 일정 주기마다 재생성합니다. Next 13 App Router에서는 이런 개념들이 **`fetch` 옵션과 Route Segment config**로 통합되었습니다. **`fetch`**를 서버 컴포넌트에서 사용할 때 **`cache: 'force-cache'`** vs **`cache: 'no-store'`** 옵션이나, **`revalidate`** 옵션으로 ISR 주기를 설정하는 방식을 알아두세요. 또한 **Prefetching** 전략 – Next.js는 Link 클릭 전에 자동 프리페치해두는 기능이 있음 – 을 이해하고, **서스펜스(Suspense)**를 통한 로딩 상태 처리 (**`<Suspense>`**와 **`loading.js`** 컴포넌트 활용)도 파악합니다.
    
    데이터 패칭과 관련해 **Axios vs React Query vs SWR**도 비교해봅니다. Axios는 단순 HTTP 클라이언트로 캐싱이나 재검증 기능이 없습니다. **React Query (TanStack Query)**는 *“복잡한 앱에서 서버 상태를 효율적으로 캐싱/갱신하는 툴”*이고, **SWR**은 *“더 가벼운 용도로 stale-while-revalidate 패턴을 구현한 Hooks”*입니다[dev.to](https://dev.to/leticiabytes/swr-vs-react-query-5el0#:~:text=SWR%20is%20a%20smaller%20library,that%20provides%20many%20more%20features)[medium.com](https://medium.com/@codeandbird/the-ultimate-guide-to-react-server-state-fetch-vs-react-query-vs-swr-b0633908194f#:~:text=The%20Ultimate%20Guide%20to%20React,revalidate%E2%80%9D%29%20is%20perfect). 예컨데 블로그/포트폴리오같이 단순 데이터는 SWR로 충분하고, 매우 복잡한 캐싱 로직이 있거나 요청 다양성이 크면 React Query가 낫다는 의견이 많습니다. 각 도구의 문서를 읽어보고, 우리 커리큘럼의 **119-123**번 항목 (RxJS 예시, tree-shaking 등) 부분은 깊이 안 파고들어도 됩니다. 대신 **React Query 혹은 SWR을 프로젝트에 도입**해보는 작은 실습을 권장합니다 – 예를 들어 SWR로 GitHub API를 호출해보고, devtools로 네트워크 요청이 어떻게 캐싱/재검증되는지 확인해보세요.
    
- **Next.js 기타 고려 사항:** 마지막으로 Next.js 앱 개발 시 알아둘 개념들을 정리합니다. **SEO 설정** – **`<head>`** 메타태그 관리, **`next/head`** 또는 App Router의 **`metadata`** export 활용 – 을 숙지하고, **동적 메타태그 (예: Open Graph)**를 경로 파라미터 기반으로 생성하는 법도 알아둡니다. **이미지 최적화** – Next.js **`<Image>`** 컴포넌트는 자동으로 다양한 크기의 이미지를 생성하고, DPR(화면 픽셀밀도)에 따라 적절한 이미지로 서빙하며, lazy loading도 지원합니다. 이에 따라 LCP 개선에 크게 도움되는 점을 기억하세요. (예: **`<Image src="/photo.jpg" width={800} height={600} alt="..." />`**를 쓰면 적절한 **`srcset`**과 캐싱이 적용됩니다.) **폰트 최적화** – **`next/font`**를 사용하면 웹폰트를 자동 preload하고 **`font-display: swap`** 등 CLS 방지 옵션도 기본
